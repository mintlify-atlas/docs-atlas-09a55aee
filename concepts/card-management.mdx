Bancard's card management system (also known as "Catastro") allows users to register their cards for future use. This enables faster checkout flows and recurring payments by storing tokenized card information.

## Overview

The card management system provides three main operations:

1. **New Card Registration:** Securely register new cards
2. **Card Retrieval:** Fetch stored cards for a user
3. **Token Management:** Handle card tokens and metadata

<Info>
Card data is stored securely by Bancard. Your application only stores tokenized references (alias tokens) that can be used for future transactions.
</Info>

### Registration Token Generation

The token for card registration uses a different concatenation pattern than Single Buy:

```python
def generate_md5_catastro(self, private_key, card_id, user_id, operation):
    # Concatenar los valores
    concatenated = f"{private_key}{card_id}{user_id}{operation}"
    
    # Generar el hash MD5
    token = hashlib.md5(concatenated.encode()).hexdigest()
    _logger.info(f"GENERO EL TOKEN CATASTRO: {token}")
    return token
```

**Location:** `models/method_new_cards.py:38-45`

### Token Components for Registration

<Steps>

<Step title="Concatenate Parameters">
Combine in this exact order:

```
private_key + card_id + user_id + operation
```

Example: `MYKEY67890123request_new_card`

Where:
- `operation` = `"request_new_card"` (constant value)
</Step>

<Step title="Generate MD5 Hash">
```python

```
</Step>

<Step title="Include in Request">
Add token to the operation object in the API payload.
</Step>

</Steps>

<Warning>
The operation string must be exactly `"request_new_card"` for new card registration. Using any other value will cause token validation to fail.
</Warning>

## Making the Registration Request

The payment provider orchestrates the card registration:

```python
def new_cards(self, user_id):
    self.ensure_one()
    
    if not all([self.clave_publica, self.clave_private]):
        raise ValidationError(
            _("El proveedor de pago Bancard no está configurado correctamente.")
        )
    
    card_id = self.env["ir.sequence"].next_by_code("bancard.card.id")
    
    method_new_cards = NewCards(
        self.clave_publica,
        self.clave_private,
        self.get_base_url(),
        card_id,
        user_id,
    )
    payload = method_new_cards.new_cards()
    response = self._bancard_make_request(payload, "/vpos/api/0.3/cards/new")
    return response.get("process_id")
```

## Card Registration Webhook

After the user completes card registration on Bancard's interface, they are redirected to your webhook:

```python
@http.route(
    "/payment/bancard/catastro/webhook",
    type="http",
    auth="public",
    csrf=False,
    methods=["POST", "GET"],
)
def bancard_webhook_catastro(self, **post):
    if request.httprequest.method == "POST":
        payment_data = request.httprequest.get_json(silent=True) or {}
        _logger.info(f"Respuesta del catastro de tarjeta: {payment_data}")
    
    elif request.httprequest.method == "GET":
        status = post.get("status")
        
        if status == "add_new_card_success":
            _logger.info("Registro Exitoso")
            return request.redirect("/catastros")
        
        elif status == "add_new_card_fail":
            _logger.info("Fallo el registro de la tarjeta")
            request.session["payment_message"] = f"{post.get('description')}"
            request.session["payment_status"] = "error"
            return request.redirect("/catastros")
```

**Location:** `controllers/page.py:9-40`

| `add_new_card_fail` | Registration failed | Show error and redirect back |

## Synchronizing Card Data

When retrieving cards from Bancard, synchronize them with your local database:

```python
# Obtener tarjetas desde Bancard
data = provider.get_cards(request.env.user.id)

# Sincronizar tarjetas en el modelo
CardModel = request.env["bancard.user.card"].sudo()
user_id = request.env.user.id

for card in data:
    existing = CardModel.search(
        [
            ("alias_token", "=", card["alias_token"]),
            ("user_id", "=", user_id),
        ],
        limit=1,
    )
    
    vals = {
        "user_id": user_id,
        "alias_token": card.get("alias_token"),
        "card_masked_number": card.get("card_masked_number"),
        "expiration_date": card.get("expiration_date"),
        "card_brand": card.get("card_brand"),
        "card_id_bancard": card.get("card_id"),
        "card_type": card.get("card_type"),
        "bancard_proccesed": card.get("bancard_proccesed", False),
    }
    
    if existing:
        existing.write(vals)
    else:
        CardModel.create(vals)
```

**Location:** `controllers/page.py:69-100`

<Tip>
Always synchronize card data when displaying the user's cards to ensure the local database reflects the current state in Bancard's system.
</Tip>

## Complete Card Registration Flow

<Steps>

<Step title="User Initiates Registration">

</Step>

<Step title="Generate Card ID">


<Step title="Create Registration Token">

<Step title="Send Registration Request">

<Step title="Handle Success/Failure">

</Step>

<Step title="Retrieve and Store Card Data">

## Security Considerations

<Warning>
**PCI Compliance:** Your application never handles raw card data. All sensitive information is entered directly on Bancard's secure pages and only tokenized references are returned.
</Warning>

### Best Practices

1. **Token Storage Only:** Never store actual card numbers or CVV codes
2. **User Association:** Always validate that cards belong to the requesting user
3. **HTTPS Required:** All card management operations must use HTTPS
4. **Private Key Security:** Keep your private key secure and never expose it client-side
5. **User Authentication:** Require authentication for all card management operations

### Access Control

```python
@http.route("/catastros/facturas", type="http", auth="user", website=True)
```

The `auth="user"` parameter ensures only authenticated users can access card management.

## Integration with Payments

Once cards are registered, you can use the `alias_token` for future transactions without requiring the user to re-enter card details. This enables:

- One-click checkout
- Recurring payments
- Subscription billing
- Saved payment methods

<Info>
Implementing payments with saved cards requires additional integration with Bancard's charge API, which uses the stored `alias_token` instead of redirecting to the payment interface.
</Info>
