The Bancard payment flow orchestrates the entire transaction lifecycle from order creation to confirmation, including webhook handling and transaction state management.

## Overview

The payment process follows a multi-step flow that involves:

1. Order preparation and process ID generation
2. Payment initiation through Bancard's Single Buy API
3. User interaction with Bancard's payment interface
4. Webhook notifications for payment confirmation
5. Order confirmation and transaction recording

<Note>
The Bancard integration supports only PYG (Paraguayan Guaraní) currency and does not require tokenization for single transactions.
</Note>

## Payment Lifecycle

<Steps>

<Step title="Order Initialization">

When a customer reaches the payment page, the system generates a unique `shop_process_id` for the transaction:

```python
shop_process_id = self.env["ir.sequence"].next_by_code("bancard.shop.process")
```

This ID is stored on the sale order and used to track the payment throughout its lifecycle.

**Location:** `models/payment_provider.py:79`

</Step>

<Step title="Single Buy Request">

The system creates a Single Buy request payload with the transaction token:

```python
def single_buy(self, monto, descripcion):
    method_single_buy = MethodSingleBuy(
        monto,
        descripcion,
        self.clave_publica,
        self.clave_private,
        self.get_base_url(),
        shop_process_id,
    )
    payload = method_single_buy.single_buy()
    response = self._bancard_make_request(payload, "/vpos/api/0.3/single_buy")
    return response.get("process_id"), shop_process_id
```

The API returns a `process_id` used to render the Bancard payment interface.

**Location:** `models/payment_provider.py:72-92`

</Step>

<Step title="User Payment Interaction">

The customer is redirected to Bancard's hosted payment page where they:
- Enter their card details
- Complete authentication (3D Secure if required)
- Confirm or cancel the payment

</Step>

<Step title="Webhook Notification (POST)">

Bancard sends a server-to-server POST notification to your webhook endpoint:

```python
@http.route(
    "/payment/bancard/webhook",
    type="http",
    auth="public",
    website=True,
    csrf=False,
    methods=["POST", "GET"],
)
def bancard_webhook(self, **post):
    if request.httprequest.method == "POST":
        payment_data = request.httprequest.get_json(silent=True) or {}
        operation = payment_data.get("operation", {})
        shop_process_id = operation.get("shop_process_id")
        description_payment = operation.get("response_description")
        
        sale_order = (
            request.env["sale.order"]
            .sudo()
            .search([("shop_process_id", "=", shop_process_id)], limit=1)
        )
```

**Location:** `controllers/main.py:22-48`

<Warning>
The webhook must return a `200 OK` response to acknowledge receipt. Failing to do so may cause Bancard to retry the notification.
</Warning>

</Step>

<Step title="Transaction Creation and Confirmation">

For approved transactions (`Transaccion aprobada`), the system:

1. Creates a payment transaction record
2. Links it to the sale order
3. Sets the transaction state to "done"
4. Confirms the sale order

```python
if description_payment == "Transaccion aprobada":
    provider = (
        request.env["payment.provider"]
        .sudo()
        .search([("code", "=", "bancard")], limit=1)
    )
    
    tx_sudo = (
        request.env["payment.transaction"]
        .sudo()
        .create({
            "provider_id": provider.id,
            "payment_method_id": payment_method_id,
            "amount": sale_order.amount_total,
            "currency_id": sale_order.currency_id.id,
            "partner_id": sale_order.partner_id.id,
            "sale_order_ids": [(6, 0, [sale_order.id])],
            "reference": sale_order.name,
            "state": "pending",
        })
    )
    tx_sudo._set_done()
    
    if sale_order.state in ("draft", "sent"):
        sale_order.action_confirm()
    
    request.env.cr.commit()
    return request.make_response("OK", status=200)
```

**Location:** `controllers/main.py:49-92`

</Step>

<Step title="User Redirect (GET)">

After completing or canceling the payment, Bancard redirects the user back to your application with a GET request:

```python
elif request.httprequest.method == "GET":
    status = post.get("status")

```

**Location:** `controllers/main.py:109-137`

</Step>

</Steps>

## Webhook Endpoints

### Payment Confirmation

- **URL:** `/payment/bancard/webhook`
- **Methods:** `POST`, `GET`
- **Authentication:** Public
- **CSRF:** Disabled

**POST (Server-to-Server):**
- Receives payment result from Bancard
- Creates transaction records
- Confirms sale orders
- Must return `200 OK`

**GET (User Redirect):**
- Handles user return after payment
- Displays success or error messages
- Redirects to appropriate page

### Cancel URL

- **URL:** `/payment/bancard/return`
- **Purpose:** User cancels payment
- **Behavior:** Redirects back to cart

**Location:** `controllers/main.py:139-153`

## Transaction States

### Approved Transaction

```json
{
  "operation": {
    "response_description": "Transaccion aprobada",
    "shop_process_id": 12345,
    "extended_response_description": "Success"
  }
}
```

Result: Order confirmed, transaction set to "done", customer redirected to confirmation page.

### Denied Transaction

```json
{
  "operation": {
    "response_description": "Transaccion denegada",
    "extended_response_description": "Insufficient funds"
  }
}
```

Result: Error message stored in session, customer redirected to payment page.

**Location:** `controllers/main.py:93-104`

## Error Handling

<Warning>
Always handle both approved and denied transactions appropriately. Store extended response descriptions to provide meaningful feedback to users.
</Warning>

### Payment Failure Flow

```python
elif description_payment == "Transaccion denegada":
    request.session["payment_message"] = (
        f"{operation.get('extended_response_description')}"
    )
    request.session["payment_status"] = "error"
```

Error messages are stored in the session and displayed when the user is redirected back to the payment page.

- Lookup of orders during webhook processing
- Rendering of the Bancard payment interface
- Transaction reconciliation

## Best Practices

<Tip>
**Always commit after confirming orders:** Use `request.env.cr.commit()` to ensure database changes are persisted before responding to Bancard.
</Tip>

1. **Idempotency:** Check if an order is already confirmed before processing webhooks
2. **Error Logging:** Log all webhook payloads for debugging and audit trails
3. **User Feedback:** Store payment errors in the session for display to users
4. **Cart Management:** Reset the shopping cart only after successful payment
5. **Return Quick Responses:** Respond to POST webhooks immediately without complex processing
